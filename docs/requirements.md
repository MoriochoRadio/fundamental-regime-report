# 요구사항 (Requirements)

> 본 문서는 1 단계 산출물. CLAUDE.md §2 (목표) + §3 (핵심 모듈) 의
> *상품적 시연 차원 명세*. 단계 2 (UX 설계) ~ 단계 6 (QA) 의 상위 제약.

---

## 1. ★ 본 시스템의 본질

> 두 표현 후보 박제 — *사용자 직접 검토 게이트에서 최종 확정 가능*
> (README 첫 문단 표현 다듬기 사이클 패턴 재사용).
> 사용자 verbatim 본질 박제 본문은 `PROGRESS.md §5.10` 사례 4호 내부 일지에
> 원본 보존.

### 1.1 본질 본문

**후보 A** (Code 권장):

> 본 시스템은 *상품적 시스템 시연* 차원의 웹 프로토타입.
> **액터**: 일반인 사용자 (ML 전문가·금융 전문가·연구자 아님).
> 연구 보고서·기술 발표 자료가 아니라 *상품성 시연 차원의 UX*.

**후보 B** (자문 다듬기):

> 본 시스템은 일반인 사용자에게 웹 페이지로 직접 시연 가능한 시스템.
> **액터**: 일반인 (ML/금융/연구 비전공).
> UX 본질은 일반인이 5 초 내 의미 파악 가능한 차원.

### 1.2 본질 위반 신호 (즉시 보고 + 정정 의무)

- 페이지에 ML 전문 용어 (PR-AUC·ROC-AUC·Brier·walk-forward·embargo 등)
  원본 노출
- 전이 행렬·log-likelihood·시드별 분산 같은 연구 차원 산출물의 *주역*
  표시
- 데이터 dump 표 또는 raw 보고서 스타일 페이지
- 일반인이 의미 파악 못 하는 시각화 (제목·축·범례가 ML 용어로만)
- 모델 카드 → 시각화 변환만 수준의 페이지 (Phase 4 의 본질 누락 패턴)

(본 1.2 항목 본문은 *§7.7 (2) self-defining 메타 박제* 차원 — 위반 신호 정의에
ML 용어 예시 인용은 논리적 필요. NFR-3 의 *UI 차원* 노출 규칙과 분리.)

### 1.3 검증 도구

각 페이지·컴포넌트는 **ML/금융 비전공 일반인이 5 초 내에 그 화면이
무엇을 보여주는지 이해 가능한가** 검증. 이 검증은 사용자 직접 검토에서
매번 적용 (§8 자기 점검 체크리스트 참조).

## 2. 사용자 페르소나

> 3 명, *모두 한국어 사용자* 가정. 모두 일반인 (ML/금융/연구 비전공).

### 페르소나 A — 산출물 평가 담당자 (기술 외 직무, 한국)

| 항목 | 값 |
|---|---|
| 배경 | 인사·평가 직무 (한국 회사 또는 외국계 한국 지사) |
| 기술 수준 | 일반 IT 능통, 학술 논문 안 읽음 |
| 본 시스템 사용 목적 | 개발자의 산출물 검토 — *무엇을 만들었고 어떤 가치가 있는가* 5 분 내 이해 |
| 핵심 기대 | "이 사람이 무엇을 만들었나, 일반인도 이해 가능한가" |

### 페르소나 B — 금융 입문 일반 투자자 (한국)

| 항목 | 값 |
|---|---|
| 배경 | 한국 일반 직장인. 주식 투자 경험 있으나 전문 분석은 안 함 |
| 기술 수준 | 모바일·웹 능통, 차트는 보지만 ML 모델은 모름 |
| 본 시스템 사용 목적 | 관심 기업 (예: 삼성전자, SK 등) 의 *재무 건강 + 시장 상황* 한눈에 이해 |
| 핵심 기대 | "이 회사 지금 어떤 상태인가, 시장은 어떤 상황인가" |

### 페르소나 C — 비전공 동료·친구 (한국)

| 항목 | 값 |
|---|---|
| 배경 | 다른 IT 분야 (백엔드·프론트·디자인) 동료, 학교 친구 |
| 기술 수준 | IT 능통 + 호기심 |
| 본 시스템 사용 목적 | 친구·동료가 만든 시스템을 *호기심*으로 둘러봄. *신기한가, 실제 동작하는가* 확인 |
| 핵심 기대 | "데모로서 매끄러운가, 들 만한 이야기가 있는가" |

## 3. 사용자 시나리오 (Use Cases)

> 6 UC. 각 UC = 페르소나 + step-by-step + 사용자 감정·기대.

### UC-1 — 첫 진입 5 초 이해 (페르소나 A 주축)

**페르소나**: A 산출물 평가 담당자 (주축).

**시나리오**: 첫 진입 5 초 내 본 시스템이 *무엇*인지 즉시 이해.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | URL 진입 | "어떤 시스템인가 빠르게 알고 싶다" |
| 2 | 첫 화면 카피 + 예시 종목 카드 확인 | "한국 KOSPI200 기업 분석 시스템이구나" 5 초 내 이해 |
| 3 | 예시 종목 카드 클릭 | "통합 화면이 어떻게 생겼는지 보고 싶다" |

- **감정**: 빠르게 본질 파악 → 호의적 평가.
- **기대 결과**: 5 초 내 *무엇을 하는 시스템* + 추가 5 초 내 *어떤 가치* 이해.

### UC-2 — 관심 기업 검색 → 통합 화면 (페르소나 B 주축)

**페르소나**: B 일반 투자자 (주축).

**시나리오**: 관심 기업 (예: 삼성전자) 의 *재무 건강 + 시장 상황* 한눈에 확인.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | 종목 검색창에 "삼성전자" 입력 | "내가 관심 있는 기업 찾기" |
| 2 | 자동완성에서 종목 선택 | "있구나" |
| 3 | 통합 화면 진입 | "이 회사 지금 어떤 상태인가" |
| 4 | 재무 건강 + 시장 상황 + 통합 리스크 카드 확인 | 색상·아이콘으로 직관 파악 |
| 5 | 해석 박스 읽기 | "왜 이런 상태인가" 한국어 자연문으로 이해 |

- **감정**: 차트·표를 *읽지 않고도* 의미 파악 → 안심.
- **기대 결과**: 30 초 내 *현 시점 종합 평가* + 1 분 내 *해석 이유* 이해.

### UC-3 — 시장 위기 시점 의미 파악 (페르소나 B 주축)

**페르소나**: B 일반 투자자 (주축).

**시나리오**: 시장 위기 시점 (코로나 등) 의 시장 상황을 직관적 파악.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | 시장 상태 화면 진입 | "전체 시장 흐름 보고 싶다" |
| 2 | 시점별 색상 시각화 확인 | "위기 시점이 색상으로 강조됨" |
| 3 | 위기 시점 hover 또는 클릭 | "이 시점은 무엇인가" 한국어 설명 |
| 4 | 일반인 해석 박스 | "위기 시점은 안전 자산 선호가 늘어남" 같은 일상 표현 |

- **감정**: 뉴스에서 본 그 시점이 *데이터로 재확인* → 신뢰.
- **기대 결과**: 1 분 내 시장 상태 흐름 + 위기 시점 의미 이해.

### UC-4 — 임의 종목 탐색 (페르소나 C 주축)

**페르소나**: C 비전공 동료 (주축).

**시나리오**: 임의 종목 (랜덤) 선택 → 통합 화면 동작 매끄러움 확인.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | 종목 검색 자동완성에서 *랜덤* 종목 선택 | "어떤 종목이든 잘 동작하나" 호기심 |
| 2 | 통합 화면 로드 시간 측정 | "빠르게 뜨네" 또는 "느리네" |
| 3 | UC-2 의 step 4-5 반복 | 임의 종목도 동일 품질 확인 |

- **감정**: *데모 매끄러움* 확인 → 시스템 품질 평가.
- **기대 결과**: 3 초 내 로드 + 어떤 종목이든 동일 UI 품질.

### UC-5 — 본 시스템 한계 안내 페이지 (페르소나 B 필수 + C 선택 + A 부수)

**페르소나**:
- **B 일반 투자자 (필수)** — 한계 페이지 정독
- **C 비전공 동료 (선택)** — 호기심 차원 확인
- **A 산출물 평가 담당자 (부수)** — UC-1 의 한계 배지로 분산 (5 분 검토 시간 가설과 한계 정독 부분 충돌 회피, 자문 정정 1)

**시나리오**: 본 시스템의 *모델 한계* 를 일반인 이해 가능 차원에서 안내.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | 첫 진입 시 한계 배너 / 모달 노출 | "이 시스템은 무엇이 한계인가" 사전 인지 |
| 2 | "한계" 페이지 진입 (B 필수) | "왜 한계인가" 깊은 이해 |
| 3 | 5 항목 한계 (일반인 표현) 확인 | "*실제 투자에 사용 금지*" 명시로 안심 |
| 4 | 한계 배지가 모든 페이지에 항상 표시 | "잊지 않고 보임" |

- **감정**: 정직성 노출 → 신뢰 ↑.
- **기대 결과**: 한계 사전 인지 + 한계 이유 깊은 이해 (B 필수) + 항상 표시
  (잊지 않음).

### UC-6 — 메뉴 둘러보기 (페르소나 A·C 공통)

**페르소나**: A 산출물 평가 담당자 + C 비전공 동료.

**시나리오**: 사이드 메뉴 둘러보기 — 5 분 내 전체 가치 이해.

| Step | 사용자 행동 | 사용자 감정·기대 |
|---|---|---|
| 1 | 사이드 메뉴 진입 | "전체 페이지 구조 파악" |
| 2 | 4~5 페이지 순회 (약 1 분/페이지) | "각 페이지 무엇을 보여주나" |
| 3 | 한계 페이지 (UC-5 의 B 필수와 다른 경로) | "이 시스템 정직한가" |

- **감정**: 전체 가치 파악 → 평가 완료.
- **기대 결과**: 5 분 내 전체 둘러보기 + 가치 + 한계 모두 이해.

## 4. 기능 요구사항 (Functional Requirements)

> 9 FR + MoSCoW 우선순위. 각 FR 의 BDD 수용 기준은 §6 에 박제.

### FR-1 — 종목 통합 화면 [**Must**]

| 항목 | 내용 |
|---|---|
| 명세 | 종목 선택 → 종목 정보 + 재무 건강 + 시장 상태 + 통합 리스크 카드 + 해석 박스 통합 표시 |
| 매핑 시나리오 | UC-2 핵심, UC-4 보조 |

| 항목 | 내용 |
|---|---|
| 명세 | 시장 상태 시점별 색상 시각화 + 위기 시점 강조 + 일반인 한국어 해석 |
| 매핑 시나리오 | UC-3 핵심 |

### FR-3 — 모델 한계 안내 [**Must**]

| 항목 | 내용 |
|---|---|

### FR-4 — 종목 검색 [Should]

| 항목 | 내용 |
|---|---|
| 명세 | 종목명 자동완성 (한국어) + 인기 종목 추천 |
| 매핑 시나리오 | UC-2 보조, UC-4 보조 |

### FR-5 — 둘러보기 가이드 [**Must**] (자문 상향)

| 항목 | 내용 |
|---|---|
| 명세 | 첫 진입 5 초 이해 카피 + 예시 종목 카드 (3 종목 권장) + 메뉴 둘러보기 가이드 |
| 매핑 시나리오 | UC-1 핵심, UC-6 핵심 |
| ★ Must 상향 사유 | **F5 는 UC-1 의 진입 게이트, F4 는 UC-2 의 분석 통로. F5 누락 시 종목 모르는 페르소나 (A·C) 가 빈 화면 응답 위험.** |

### FR-6 — 두 기업 비교 [Could]

| 항목 | 내용 |
|---|---|
| 명세 | 두 기업 나란히 비교 화면 |
| 매핑 시나리오 | UC-2 확장 (향후) |

### FR-7 — LLM 서술 해석 표시 [Should]

| 항목 | 내용 |
|---|---|
| 명세 | LLM 빌드타임 배치 서술 (한국어 자연문) 의 표시. *런타임 LLM 호출 0* (CLAUDE.md §3.4) |
| 매핑 시나리오 | UC-2 해석 박스 보강 |

### FR-8 — 실시간 데이터 [**Won't**]

| 항목 | 내용 |
|---|---|
| 명세 | 실시간/라이브 데이터 처리 *금지* |
| 사유 | CLAUDE.md §4.2 직접 매핑. 본 프로젝트는 배치 분석 |

### FR-9 — ML 전문 차원 화면 [**Won't**]

| 항목 | 내용 |
|---|---|
| 명세 | 전이 행렬·시드별 분산·log-likelihood 같은 *연구 차원 산출물의 주역 표시 화면* 신설 금지 |
| 사유 | §1 본질 위반 신호 직접 매핑 — 일반인 5 초 이해 차원과 충돌 |
| 비고 | 동일 정보의 *모델 카드* 노출은 별도 차원 (`reports/*.md` 기술 차원 문서, 사용자 명시 허용) |

## 5. 비기능 요구사항 (Non-Functional Requirements)

> 9 NFR + MoSCoW + 수치 평이.

### NFR-1 — 성능 [**Must**]

| 항목 | 수치 |
|---|---|
| 페이지 초기 로딩 | < 3 초 (Chrome DevTools Performance) |
| 차트 렌더 | < 1 초 (단일 차트) |
| 종목 전환 | < 1 초 (캐시 이후) |
| 사유 | Phase 4 vacuous 의 *렉* 문제 직접 매핑. 5 초 일반인 이해 차원의 *초기 인상* 결정 |

### NFR-2 — 사용성 / 일반인 5 초 검증 [**Must**]

| 항목 | 수치 |
|---|---|
| 페이지·컴포넌트 의미 5 초 이해 | 일반인 N 명 (≥ 3) 직접 검토에서 통과 |
| 검증 방법 | 사용자 직접 검토 게이트 (단계 6 QA 의무) |
| 사유 | §1 본질 박제의 핵심 검증 도구 |

### NFR-3 — 사용성 / UI ML 용어 원본 노출 0 [**Must**]

| 항목 | 수치 |
|---|---|
| UI 페이지·컴포넌트에 ML 전문 용어 원본 노출 | 0 건 (자동 검사 가능) |
| 검증 방법 | grep 검사 — UI 코드 + 렌더링 결과 (예: PR-AUC, ROC-AUC, Brier, walk-forward, embargo, HMM, K-Means, log-likelihood, ablation, fold 등) |
| 스코프 | **UI 차원만**. 본 `docs/requirements.md` + `reports/*.md` 기술 차원 문서는 *NFR-3 적용 외* (§1.2 self-defining 메타 박제 분리) |
| 사유 | §1 본질 위반 신호 #1 직접 매핑 |

### NFR-4 — 모델 한계 안내 [**Must**]

| 항목 | 수치 |
|---|---|
| 항상 표시 배지 | 모든 페이지 1 회 이상 |
| 첫 진입 1 회 모달 | 세션 첫 진입 시 1 회 |
| Limitations 페이지 | 1 페이지 존재 + 메뉴 노출 |
| "*실제 투자에 사용 금지*" 명시 | 배지·모달·페이지 모두 |
| 사유 | FR-3 와 직접 매핑 — 정직성 사슬의 UX 차원 |

### NFR-5 — 정확성 [**Must**]

| 항목 | 수치 |
|---|---|
| UI 표시 수치 = 모델·데이터 산출물 일치 | 100% (사후 변경 0) |
| 검증 방법 | 단계 6 QA — UI 수치 vs `data/interim/`·`reports/` 산출물 일치 확인 |
| 사유 | 정직성 사슬 자산 보존 |

### NFR-6 — 한국어 [**Must**]

| 항목 | 수치 |
|---|---|
| 모든 UI 한국어 | 100% (영어 보조 단어 가능 — 예: KOSPI200 같은 고유명) |
| 사유 | 페르소나 A·B·C 모두 *한국어 사용자* 가정 명시 |

### NFR-7 — 색맹 [Should]

| 항목 | 수치 |
|---|---|

| 검증 방법 | 색맹 시뮬레이터 (예: Chrome DevTools Rendering) |

### NFR-8 — 반응형 [Should]
