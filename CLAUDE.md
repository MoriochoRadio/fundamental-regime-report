## 1. 프로젝트 정체성

- **제목(가안)**: 기업 펀더멘털 + 시장 국면 인지형 통합 분석 리포트 시스템
- **성격**: 1인 개발자의 **취업 포트폴리오 프로젝트**
- **도메인 이력**: 직전 프로젝트는 AI 기반 의료영상 분석(객체탐지·도메인 적응) — 웹 구현 + GitHub 형상관리 + 논문 초안까지 완료. 본 프로젝트는 동일한 역량을 **금융 도메인**으로 이전하기 위한 작업이다.

---

## 2. 목표 (Goal)

사용자가 특정 기업을 선택하면,

1. 그 기업의 **재무 건전성·리스크 분석 리포트**를 자동 생성하고,
2. 동시점의 **시장 국면(regime)** 및 해당 기업의 **주가 궤적**을
3. 하나의 **통합 대시보드**에서 함께 보여준다.

**차별화 포인트**: 펀더멘털 리스크 해석을 **시장 국면에 조건부**로 제시한다.
(예: "동일한 부채비율 상승이라도, 위험회피 국면에서는 더 강하게 해석한다.")

### 3.4 ★ AI/ML 역할 분리 (Portfolio Identity)

본 프로젝트가 *직접 학습·개발하는 AI* 는 다음 두 가지이며, 이 두 가지가
포트폴리오 평가의 **기둥**이다:

- 펀더멘털 **부실/등급 스코어링 모델** (지도학습; sklearn + LightGBM)
- 시장 **국면 분류 모델** (비지도학습; hmmlearn 본 라인, GMM/K-Means 비교)

LLM은 *직접 만든 AI* 가 **아니다**. LLM은 *이미 검증된 수치·라벨을
한국어로 서술하는 보조 도구*다. 따라서:

- LLM은 어떤 수치·라벨도 **결정하지 않는다**. 입력으로만 받는다.
- LLM 출력은 *재무 사실의 새 생성이 아니라 기존 사실의 변환*이다.
- LLM 공급자는 부품이며, `LLMProvider` 인터페이스로 추상화하여
  **교체 가능**해야 한다.
- LLM 호출은 **빌드 타임 배치**에서 1회만 발생한다. **서비스 런타임
  (대시보드)는 LLM 호출 0회** — 정적 산출물(JSON)만 읽는다.

## 5. 방법론적 원칙 (Methodological Rigor)

본 프로젝트는 포트폴리오로서 **방법론적 엄밀성**을 통해 평가될 것을
가정한다. 다음 원칙은 모든 모델링 단계에 적용된다.

- **데이터 누수 차단 (No data leakage)**: 학습 시점 이후의 정보는
  어떤 형태로도 학습 피처에 들어가지 않는다.
- **룩어헤드 차단 (No look-ahead bias)**: 재무제표는 **공시일 + 안전한
  지연(lag)** 이후부터 사용 가능하다고 간주한다. 시장 국면 라벨도
  시점 t 이전 데이터로만 산출되어야 한다.
- **시간 인지 분할 (Time-aware splits)**: 전통적 무작위 K-Fold 금지.
  Walk-forward / expanding window 사용.
- **재현 가능성**: 모든 데이터·모델 산출은 시드 고정 + 입력 스냅샷 기준
  재현 가능해야 한다.
- **국면 분류 결과의 사후 해석**: 국면은 사후적으로 명명될 수 있으나,
  국면 **라벨 자체는 사전적(causal)** 으로 산출되어야 한다.
- ★ **유니버스 변수의 모델 피처 격리**: 유니버스 정의에 쓰이는 변수
  (KOSPI200 편입/편출 여부·이벤트, 시총 순위, 인덱스 비중 등)는
  **펀더멘털 모델의 피처로 사용 금지**다. 이런 변수는 *살아남음의
  함수*이므로 피처로 들어가면 *데이터 누수 회로*가 형성된다
  (예: "유동성 큰 종목 = 부실 적음"). 유니버스 정의는 `src/frr/universe.py`
  에 격리하고, 피처 모듈은 이 정보에 접근하지 않는다.
- ★ **상장폐지/관리종목 메타데이터의 모델 피처 격리**: 상장폐지일
  (`DelistingDate`), 상장폐지 사유(`Reason`), 관리종목 지정일
  (`ArrantEnforceDate/EndDate`) 등은 **부실 라벨(D2) 정의에만 사용**하고,
  **펀더멘털 모델의 피처로 절대 흘러들어가서는 안 된다**. 이들은
  *결과*이지 *원인*이 아니므로 피처로 들어가면 라벨 누수(label leakage)다.
  단계 2 DoD에 *상장폐지 메타데이터 격리 검증 테스트*를 둔다.
- 위 두 격리 원칙은 단계 2 진입 시 *룩어헤드 차단 테스트*와 함께
  검증 항목으로 묶여 실행된다.

본 섹션은 어시스턴트(Claude)와의 협업 방식 자체에 대한 **고정 규칙**이다.

### 7.1 문서 운영
- **CLAUDE.md (이 파일)**: 변하지 않는 사실·규칙·방향만 담는다.
  새 규칙·결정이 확정되면 반영한다.
- **PROGRESS.md**: 작업 단위가 끝날 때마다
  *현재까지 완료 / 현재 상태 / 다음 할 일 / 미해결 이슈* 를 갱신한다.

### 7.2 코드 작성 전 절차
- 코드를 바로 짜지 않는다.
- 다음 항목은 **사용자 확인 후** 진행한다:
  - 권장 기술 스택
  - 디렉터리 구조
  - 단계별 진행 계획
  - 중요한 설계 결정(데이터 모델, 모델 선택, 평가 지표, 리포트 포맷 등)

### 7.3 단계 정의 (큰 흐름)
1. 데이터 셋업 (DART / pykrx 수집·정제·캐싱)
2. 펀더멘털 모듈 (스코어링 + LLM 리포트)
3. 국면 모듈 (HMM/클러스터링)
4. 통합 대시보드
5. 마무리 (문서·재현성·발표 자료)

각 단계는 명확한 **완료 기준(DoD)** 을 가져야 하며, 단계 종료 시
PROGRESS.md를 갱신하고 git 커밋을 제안한다.

### 7.4 커밋
- 한 작업 단위가 끝날 때마다 **단위가 명확한 git 커밋**을 제안한다.
- 커밋 메시지는 *무엇을 / 왜* 가 드러나야 한다.
- **모든 코드 변경 커밋 전 `uv run ruff format src/frr tests scripts`
  실행이 필수다.** 이 규칙은 2026-05-18 의 CI 4회 연속 실패
  (`ruff format --check` 누락) 의 재발 방지 차원.

  - 분석 대상 유니버스 확정 (KOSPI200 vs 다른 집합)
  - 부실 라벨 정의 (관리종목 지정? 등급 다운그레이드? 신용 이벤트?)
  - 국면 개수와 모델(HMM의 상태 수, 클러스터링 알고리즘 등)
  - 대시보드 프레임워크 (Streamlit / Dash / Next.js 등)

  - **KRX 정보데이터시스템 분기 CSV (수동 다운로드)** — 시점별
    KOSPI200 구성. `data/external/kospi200_quarterly/` 에 저장,
    git 추적, 매니페스트와 함께.
- 시장 범위: **한국 단일 시장 (KOSPI200 한정)**
- 분석 기간: **2015-01-01 ~ 2024-12-31** (가용성 사유, §4.1)
- 분석 방식: **배치(과거 데이터)**

- CI: **GitHub Actions** (lint + test만)

### 8.3 데이터·산출물
- 데이터 처리: **pandas** + **numpy** (대용량 시 polars 부분 도입)
- 저장 포맷: **parquet (Snappy)** + JSON
- 산출물 추적: 파일시스템 + git (MLflow 미도입)
- 재현성: seed 고정, 입력 스냅샷 해시, 시간 인지 분할 명시
- **외부 데이터 매니페스트 정책**: `data/external/` 의 모든 외부 파일
  (수동 다운로드 CSV 등)은 옆에 매니페스트(`MANIFEST.yaml` 또는
  파일별 sidecar)를 둔다. 각 항목은 최소한 다음을 기록한다:
  - `source` (URL 또는 시스템명: 예 "KRX 정보데이터시스템")
  - `reference_date` (기준일자, YYYY-MM-DD)
  - `downloaded_at` (수집 시점, YYYY-MM-DD)
  - `download_path` (KRX 메뉴 경로 등 재수집 방법)
  - `sha256` (파일 해시, 변조 감지)
  - `notes` (필요 시)
  매니페스트도 git 추적한다. *재현성과 추적가능성*은 외부 데이터의
  필수 조건이다.
- **줄바꿈 정규화 정책 (sha256 무결성)**: CSV 등 외부 텍스트 파일은
  `.gitattributes` 에서 `text eol=lf` 로 지정해 *모든 OS·워크트리에서
  LF 로 통일* 한다. `binary` 매크로는 사용 금지 — *워크트리 CRLF 를
  그대로 index 로 propagate* 해 매니페스트 sha256 을 깨뜨린다 (2026-05-19
  실측 확인, PROGRESS §4-pre). 매니페스트의 `sha256` 도 **git index 바이트
  (LF 정규화 후) 기준** 으로 계산해야 OS·체크아웃 환경과 무관하게 재현
  가능하다.

### 8.4 모델

### 8.5 프레젠테이션

- 시각화 (대화형): **plotly**
- 시각화 (정적·리포트): **matplotlib**

### 8.6 레이아웃 원칙
- **src layout**: 메인 패키지는 `src/frr/` (이름은 `frr`).
  설치 후 `from frr.data import dart` 형태로 임포트.
- **LLM 단일 출구**: 외부 LLM SDK는 **오직 `src/frr/llm/`** 에서만
  import한다. 다른 모듈은 `LLMProvider` 인터페이스만 본다.
- **`app/` 정적 읽기 전용**: Streamlit 코드는 어떤 학습·계산·LLM 호출도
  하지 않는다. `reports/` 의 정적 JSON과 `models/`의 학습된 모델만 읽는다.
  → CI에서 `app/` 안에 LLM SDK import가 없는지 검사 가능.
- **데이터 흐름 단방향**: `raw → interim → processed → reports`.
  역방향 의존(예: processed가 raw를 다시 패치) 금지.
- **점진 생성**: 디렉터리·파일은 *단계 진입 시점*에 필요한 만큼만 만든다.
  빈 패키지나 placeholder 파일을 미리 두지 않는다.

## 9. 안티-패턴 (Do Not)

- "이것도 해두면 좋을 것 같아서" — 범위 확장 금지.
- "어차피 비슷하니까" — 시간 인지 분할 없이 평가 금지.

  단에 있다. LLM은 **이미 검증된 수치의 서술화**를 맡는다.

- "대시보드에서 LLM을 바로 부르면 더 자연스럽지" — 금지. 서비스 런타임
  LLM 호출 0회 원칙(§3.4)을 깨면 비용·재현성·평가 모두 무너진다.
